# CLAUDE.md — הקשר פרויקט למערכת ניהול מועדוני ספורט

> קובץ זה נקרא אוטומטית ע"י Claude Code בכל סשן. הוא מסכם את הכללים הקריטיים.
> המסמך המלא: `club-saas-spec.md` (קרא אותו לפני עבודה על תחום חדש).

## מה אנחנו בונים

מערכת SaaS multi-tenant לניהול מועדוני ספורט (CMS תפעולי). שתי רמות גישה:
- **Control Plane** — פאנל-על שלי (Platform): יצירת מועדונים, מנויים, דשבורד רוחבי.
- **Tenant Plane** — ה-CMS שהמועדון משתמש בו: שחקנים, קבוצות, עונות, תשלומים.

קהל יעד: ישראל. עברית, **RTL**, מע"מ וחשבוניות מקומיות.

## Stack (הוכרע — אל תסטה בלי לשאול)

- DB: PostgreSQL דרך **Supabase**
- Auth: **Supabase Auth** (מייל+סיסמה, OTP ב-SMS)
- Frontend: **Next.js + React + Tailwind + shadcn/ui**
- Storage: Supabase Storage (לוגואים)
- לוגיקת כסף: Supabase Edge Functions / Next.js API routes
- Hosting: Vercel + Supabase

## כללי ברזל (לא לשבור — אלה ההחלטות היקרות לתיקון בדיעבד)

1. **Multi-tenancy:** DB משותף. כל טבלה רלוונטית מכילה `club_id`. בידוד נאכף ב-**RLS**. אסור להסתמך רק על `WHERE` בקוד.
2. **Scoping דו-שכבתי:** זהות (player, coach, contact) קבועה עם `club_id` בלבד. שיוך (team_players, team_coaches) עונתי עם `club_id` + `season_id`. אסור לשבץ `season_id` בתוך ישות הזהות.
3. **Authorization:** RLS הוא רשת ביטחון אחרונה, לא הקו הראשון. כל endpoint נבדק מפורשות. **default-deny.**
4. **אנשי קשר (Contact) = נתונים בלבד.** אין להם login. רק staff מקבלים `User`. אסור לקשר Contact ל-User.
5. **כסף באגורות (integer), לא float.** מטבע מפורש בכל רשומת כסף. שיעור מע"מ נשמר היסטורית על כל חשבונית.
6. **תשלומים:** עמוד סליקה מתארח (hosted) בלבד — לעולם לא נוגעים בפרטי כרטיס (PCI). webhooks עם אימות חתימה + idempotency (טבלת webhook_log). חיוב = state machine, לא בוליאני שולם/לא.
7. **Soft-delete** (`deleted_at`), לא מחיקה פיזית. שחקן שעזב = סטטוס, לא נמחק.
8. **Audit log** לכל פעולה רגישה (כסף, הרשאות, גישת תמיכה לתוך מועדון).
9. **Platform users** = טבלה נפרדת לגמרי מ-`users` של המועדונים.

## סדר העבודה — לפי ה-roadmap, שלב אחר שלב

אל תבנה הכל בבת אחת. בנה Phase אחר Phase, כל אחד נבדק לפני שבונים עליו:

- **Phase 0** — תשתית: Supabase, סכמת DB + RLS, Auth, דשבורד פלטפורמה מינימלי, audit log, queue, גיבויים.
- **Phase 1** — ליבה: עונות, קבוצות, שחקנים, מאמנים, גלגול עונה, branding, דשבורד מועדון.
- **Phase 2** — תשלומים: Contacts, payment_plans, חיובים, סליקה (webhooks+idempotency), קישורי תשלום, reconciliation.
- **Phase 3** — RBAC מלא, הזמנת staff, audit log גלוי.
- **Phase 4** — אנליטיקה, פעילויות, subscription billing.

## עקרונות עבודה

- עברית RTL בכל ה-UI מההתחלה — לא "נוסיף אחר כך".
- לכל feature שנוגע בכסף או הרשאות — כתוב בדיקה שמוודאת בידוד בין מועדונים.
- שאל לפני סטייה מהכללים לעיל. הם לא שרירותיים.

## תקני הנדסה — כתוב קוד כמו מפתח בכיר (עשרות שנות ניסיון)

אתה מהנדס תוכנה בכיר. הקוד שאתה כותב חייב להיות נקי, מאורגן, קל לתחזוקה, ומחולק נכון. **עדיף לעצור ולתכנן מבנה לפני שכותבים** מאשר לייצר קוד מהר ולתקן אחר כך.

### גודל קבצים ופירוק
- **קובץ מעל ~300 שורות = סימן אזהרה.** פצל אותו. אין קבצים של 1000+ שורות, לעולם.
- פונקציה מעל ~40 שורות — שקול לפצל. פונקציה עושה **דבר אחד**.
- רכיב React מעל ~150 שורות — פצל לרכיבים קטנים יותר.
- קובץ = אחריות אחת. אם קשה לתת לקובץ שם מדויק, הוא עושה יותר מדי.

### מבנה תיקיות (feature-based, לא type-based)
ארגן לפי **תחום עסקי**, לא לפי סוג טכני. לא `controllers/`, `services/`, `models/` ענקיים — אלא:
```
src/
  app/
    (control-plane)/   ← האזור שלי (admin): יצירת מועדונים, מנויים
    (tenant)/          ← האזור של המועדונים (app): שחקנים, תשלומים...
  features/            ← לוגיקה משותפת לפי תחום
    players/      (הכל על שחקנים: ui, logic, queries, types)
    teams/
    payments/
    seasons/
    contacts/
  lib/            (תשתית משותפת: supabase client, auth, utils)
  components/ui/  (רכיבי shadcn משותפים)
  types/          (טייפים גלובליים)
```
כך קל למצוא דברים, וכל feature הוא יחידה עצמאית.

**הפרדת שני האזורים (קריטי):** Control Plane (שלי) ו-Tenant Plane (מועדונים) הם **אזורים נפרדים באותו פרויקט** — לא שני פרויקטים. ההפרדה:
- routing נפרד: `(control-plane)` מול `(tenant)`, עם layouts ו-auth נפרדים.
- auth נפרד: `platform_users` (שלי) מול `users` (מועדונים) — לעולם לא מתערבבים.
- אזור Control Plane אינו מסונן ל-club_id יחיד; אזור Tenant תמיד כן (RLS).
- קוד/DB/מודלים **משותפים** — אל תשכפל. רק שכבת הגישה והגבול נפרדים.

### עקרונות קוד
- **DRY** — אל תשכפל לוגיקה. חזרה שלישית על אותו דבר = הוצא לפונקציה/hook משותף.
- **Separation of concerns** — לוגיקה עסקית נפרדת מ-UI נפרד מגישה ל-DB. רכיב React לא מכיל שאילתת SQL גולמית; הוא קורא ל-data layer.
- **Single source of truth** — כל טיפוס/קבוע מוגדר במקום אחד ומיובא משם.
- **Naming מפורש** — שמות משתנים ופונקציות שמסבירים את עצמם. `getActiveSeasonPlayers` ולא `getData`. בלי קיצורים מעורפלים.
- **Early return** על תנאים, לא קינון עמוק של if-ים.
- **TypeScript קפדני** — `strict: true`. אסור `any` בלי הצדקה. טייפים מפורשים על גבולות (פונקציות, API).

### Exports ו-imports
- **Named exports** כברירת מחדל (לא default export), חוץ מ-pages/רכיבי Next שדורשים default.
- כל feature מייצא דרך `index.ts` (barrel file) — מי שמייבא מבחוץ מייבא מה-index, לא מקבצים פנימיים. כך הגבול הציבורי של ה-feature ברור.
- אל תייבא בין features ישירות לקבצים פנימיים — רק דרך ה-public API שלהם.

### Data layer (קריטי לפרויקט הזה)
- **כל גישה ל-DB עוברת דרך שכבת data ייעודית** לכל feature (למשל `features/players/queries.ts`). אסור שאילתות Supabase מפוזרות ברכיבי UI.
- כל שאילתה תמיד מסוננת ל-`club_id` (גם אם RLS מגבה — הגנת עומק).
- טיפוסים של ה-DB מיוצרים אוטומטית מ-Supabase (`supabase gen types`) — לא כותבים ידנית.

### איכות ותחזוקה
- שגיאות מטופלות מפורשות — לא בולעים errors. כסף והרשאות: כשל קולני, לא שקט.
- אין magic numbers/strings — קבועים בעלי שם.
- הערות מסבירות **למה**, לא **מה** (הקוד אמור להסביר את ה-מה בעצמו).
- כל מודול שנוגע בכסף/הרשאות/בידוד — מלווה בבדיקות.
- ESLint + Prettier מוגדרים ונאכפים מהיום הראשון.

### לפני שכותב feature חדש
1. תכנן את המבנה (אילו קבצים, מה כל אחד עושה) — והצג לי לפני שמתחיל.
2. הגדר את הטיפוסים והגבולות קודם.
3. כתוב את ה-data layer, ואז הלוגיקה, ואז ה-UI.
4. אל תכתוב 500 שורות בלי לעצור. בנה במנות קטנות וניתנות לבדיקה.

## תהליך עבודה ואמינות (כך עובד מפתח בכיר)

### Git ו-commits
- commits קטנים ואטומיים — כל commit עושה דבר אחד שלם וניתן לתיאור במשפט.
- הודעות commit ברורות בפורמט conventional: `feat:`, `fix:`, `refactor:`, `test:`, `chore:`.
- אל תערבב refactor עם feature באותו commit.
- branch לכל feature — לא עובדים ישירות על main.

### כשלא בטוח — עצור ושאל
- אם דרישה לא ברורה או יש כמה דרכים לממש — **שאל לפני שכותב**, אל תנחש.
- אם אתה עומד לסטות מ-CLAUDE.md או מהמסמך — עצור, הסבר למה, וקבל אישור.
- אם משהו במסמך נראה לך שגוי או לא עקבי — ציין זאת, אל תתקן בשקט.
- עדיף שאלה אחת טובה מאשר שעת עבודה בכיוון הלא נכון.

### בדיקות (Testing)
- כל לוגיקה שנוגעת ב**כסף, הרשאות, או בידוד בין מועדונים** — חייבת בדיקות. זה לא אופציונלי.
- בדיקת בידוד חובה: ודא שמשתמש ממועדון א' לא יכול לקרוא/לכתוב נתוני מועדון ב' — גם דרך ה-API וגם דרך ה-DB.
- בדוק את ה-happy path וגם את מקרי הקצה (תשלום נכשל, webhook כפול, עונה סגורה).
- אל תסמן משימה כ"הושלמה" לפני שהבדיקות עוברות.

### אבטחה וסודות (Secrets)
- **לעולם אל תכתוב secrets בקוד** — מפתחות API, connection strings, סיסמאות. הכל ב-environment variables.
- ודא ש-`.env` ב-`.gitignore` מהרגע הראשון. אל תעשה commit לסודות.
- מפתחות סליקה: sandbox ב-dev, production רק ב-production — לעולם לא מעורבבים.
- ולידציה על כל input מהמשתמש (schema validation, למשל Zod). אל תסמוך על ה-frontend.

### אחרי כל שלב
- ודא שהקוד מתקמפל ושאין שגיאות lint/type לפני שממשיכים.
- אם שינית משהו שעלול לשבור קוד קיים — בדוק שלא שברת.
- סכם בקצרה מה נבנה ומה השלב הבא.

### Definition of Done (מתי feature באמת גמור)
פיצ'ר גמור רק כאשר: הקוד עובד ✓ | מחולק נכון לקבצים ✓ | TypeScript נקי ✓ | lint נקי ✓ | בדיקות עוברות (אם נוגע בכסף/הרשאות) ✓ | RTL תקין ✓ | אין secrets בקוד ✓

## מערכת עיצוב (Design System)

השתמש ב-design tokens המוגדרים בקובץ `design-system.md` (אם קיים בפרויקט) או בהגדרות Tailwind. **אל תמציא צבעים/מרווחים ad-hoc** — השתמש ב-tokens. כל רכיב חדש עקבי עם מערכת העיצוב: אותם רכיבי כפתור/אינפוט/כרטיס, אותו ריווח, אותה טיפוגרפיה. עקביות ויזואלית > יצירתיות נקודתית.
