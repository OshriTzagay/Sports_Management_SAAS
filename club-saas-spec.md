# מסמך אפיון — מערכת ניהול מועדוני ספורט (SaaS)

> מסמך עבודה חי. הכוונה: בסיס יציב ל-MVP שניתן להרחיב בלי לשבור את הארכיטקטורה. כל החלטה מהותית מסומנת ומנומקת כדי שתבין *למה*, לא רק *מה*.

---

## 1. סקירה כללית

מערכת SaaS מבוססת ענן לניהול מועדוני ספורט (כדורגל ואחרים). המערכת היא בעצם **CMS תפעולי** שמאפשר למועדון לנהל את כל ההיבטים: שחקנים, קבוצות, עונות, אנשי קשר ותשלומים, הרשאות ואנליטיקה.

המערכת בנויה כ**מוצר אחד עם שתי רמות גישה** (לא שני פרויקטים נפרדים):

1. **Control Plane (אתה — Platform)** — פאנל-על פנימי ליצירת מועדונים (tenant provisioning), ניהול מנויים (subscriptions), ומעקב על בריאות המערכת. מעט משתמשים, גישה רוחבית לכל המועדונים.
2. **Tenant Plane (הלקוח — Club App)** — ה-CMS עצמו שאליו נכנס המועדון אחרי שיצרת אותו. כל בקשה כאן *תמיד* מסוננת ל-`club_id` יחיד. כאן מתרחשת כל העבודה היומיומית.

הפירוט המלא על ההפרדה בין השכבות בסעיף 2.4.

קהל יעד: ישראל. עברית, RTL, מע"מ וחשבוניות מקומיות.

## 2. החלטות ארכיטקטורה מרכזיות (ונימוקים)

### 2.1 Multi-Tenancy: DB משותף + `club_id` + Row-Level Security

**ההחלטה:** בסיס נתונים אחד משותף. כל טבלה רלוונטית מכילה עמודת `club_id`. הבידוד נאכף ברמת ה-DB באמצעות **PostgreSQL Row-Level Security (RLS)**.

**למה לא DB נפרד לכל מועדון:**

- **תחזוקה:** כל migration היה צריך לרוץ על N בסיסי נתונים בנפרד. כשלון באמצע משאיר את הסכמות במצבים שונים — סיוט תפעולי שגדל ליניארית עם מספר הלקוחות.
- **עלות:** N instances/connections עולים כסף, במיוחד כשמועדוני ספורט רדומים רוב השנה (עונתיות).
- **בידוד מושג אחרת:** RLS אוסר ברמת ה-DB על שורה להיחשף אם `club_id` לא תואם להקשר הבקשה. גם אם מפתח שכח `WHERE club_id = ?` בקוד — ה-DB עצמו חוסם דליפה.

**Escape hatch:** אם בעתיד יגיע לקוח enterprise/ענק שדורש בידוד פיזי — אפשר "לשלוף" אותו ל-DB משלו בלי לשנות את כל הארכיטקטורה. מתחילים פשוט.

### 2.2 שכבת ה-Scoping: לא רק `club_id` — גם `season_id`

זו תובנה קריטית למודל הזה. מועדון ספורט הוא ישות **עונתית**. כמעט כל ישות "חיה" (שיוך שחקן לקבוצה, חיובים, נוכחות) שייכת לעונה ספציפית.

**הכלל המנחה:**

| סוג ישות | Scope |
|---|---|
| ישויות ליבה קבועות (פרטי שחקן, פרטי איש קשר, פרטי מאמן, פרטי מועדון) | `club_id` בלבד |
| ישויות עונתיות (שיוך לקבוצה, תשלום, נוכחות, סטטיסטיקה) | `club_id` + `season_id` |

**למה זה חשוב:** שחקן *קיים* פעם אחת במערכת (זהותו, ת.ז., אנשי הקשר שלו). אבל ה*שיוך* שלו לקבוצה משתנה מעונה לעונה. אם תשבץ `season_id` בתוך טבלת השחקן עצמו — תיצור כפילויות ותאבד את ההיסטוריה. ההפרדה הזו מאפשרת לך לשאול "מי שיחק ב-2023?" מול "מי משחק עכשיו?" בלי לאבד נתונים.

**ברירת מחדל בממשק:** הפילטר תמיד נפתח על **העונה הפעילה הנוכחית**, עם אפשרות החלפה לעונות עבר (קריאה בלבד מומלצת לעונות סגורות).

### 2.3 מה עוד מקובל לעשות ב-SaaS (וכדאי לשלב מההתחלה)

שאלת — אז הנה דברים שחוסכים כאב בהמשך:

- **Soft Delete** (`deleted_at`) במקום מחיקה פיזית. מאפשר שחזור, audit ועמידה בדרישות פרטיות. מחיקה אמיתית רק ב-job מתוזמן.
- **Audit Log** — מי עשה מה ומתי. קריטי כשמדובר בכסף והרשאות. "מי שינה את הסכום?" חייב להיות לו מענה.
- **Subscription / Billing למודל ה-SaaS עצמו** — נפרד לגמרי מהסליקה של ההורים. זה איך *אתה* גובה מהמועדון (תוכניות, מגבלות, trial). אפשר לדחות ל-Phase 2 אבל לתכנן את הטבלאות מראש.
- **Feature Flags / Plan Limits** — "מועדון בתוכנית Basic יכול עד 3 קבוצות". מאפשר tiered pricing בלי שינוי קוד.
- **Idempotency** בכל מה שנוגע לכסף — מניעת חיוב כפול כש-webhook מגיע פעמיים.
- **Background Jobs / Queue** — שליחת SMS, מיילים, תזכורות תשלום. לא לעשות בתוך ה-request.
- **Notifications כתשתית, לא feature מפוזר** — SMS/מייל/push חוזרים בכל המערכת (קישור תשלום, תזכורת, הזמנת משתמש, קבלה). שכבת notifications אחת עם templates ו-providers מתחלפים (ספק SMS, ספק מייל) — לא קוד שליחה משוכפל בכל מקום. גם תשתית להעדפות (מי מסכים לקבל מה).
- **Data Retention ופרטיות קטינים** — נתוני קטינים כפופים לחוק הגנת הפרטיות. הגדר מראש כמה זמן שומרים נתונים אחרי ששחקן עוזב, מנגנון "זכות להישכח" (מחיקה לפי בקשה), ומינימום נתונים שנאספים. עדיף לתכנן מההתחלה מאשר לרדוף אחרי זה.
- **Environment Separation** — dev / staging / production נפרדים לחלוטין, כולל מפתחות סליקה נפרדים (sandbox מול production).
- **Observability, Alerting & Rate Limiting** — מוצר בתשלום שנופל בשקט מאבד לקוחות. צריך מההתחלה: (1) **logging מובנה** (structured logs עם `club_id`/`request_id` להקשר), (2) **error tracking** (Sentry או דומה) שתופס חריגות בזמן אמת, (3) **health checks + uptime monitoring** על שני האזורים, (4) **alerting** על אירועים שחובה לדעת עליהם מיד — שגיאות 5xx, כשלי webhook מהסולק, כשלי תשלום חריגים, ירידת זמינות, (5) **rate limiting** על endpoints רגישים (login, OTP, payment, webhook) כהגנה מ-brute-force ו-abuse. זה לא "אחר כך" — בלי זה אתה עיוור לתקלות אצל לקוח משלם.

### 2.4 הפרדת השכבות: Control Plane מול Tenant Plane

יש לך לכאורה "שתי מערכות", אבל מבחינה ארכיטקטונית זו **רמת גישה אחת מעל השנייה** על אותו מוצר ואותו DB:

| היבט | Control Plane (אתה) | Tenant Plane (מועדון) |
|---|---|---|
| מי | Platform Admin / Support | Club Admin, Staff, אנשי קשר |
| גישה ל-`club_id` | רוחבית — כל המועדונים | תמיד מסונן ל-`club_id` יחיד (RLS) |
| UI / Domain | `admin.yourapp.com` | `app.yourapp.com` (או subdomain למועדון) |
| Auth | טבלת `platform_users` נפרדת | טבלת `users` של המועדונים |

**שלוש החלטות קריטיות:**

- **DB משותף, סינון שונה.** ה-Control Plane רואה את כל המועדונים; ה-Tenant Plane תמיד מסונן ל-`club_id` יחיד דרך RLS. אותו בסיס נתונים, שתי זוויות.
- **אל תהפוך את Platform Admin ל-role בתוך users של מועדון.** הוא ישות נפרדת לגמרי (`platform_users`) עם auth נפרד. ערבוב = יום אחד באג חושף את כל המועדונים.
- **גישת תמיכה ("היכנס בתור המועדון") היא פעולה מפורשת ומתועדת** ב-audit log — לא דלת אחורית קבועה שעוקפת RLS.

### 2.5 טיפול בכסף (החלטות שחובה לסגור מראש)

- **כל הסכומים נשמרים באגורות (integer), לא בשקלים עם נקודה עשרונית.** חישובי כסף ב-float הם מקור באגים קלאסי (0.1 + 0.2 ≠ 0.3).
- **מטבע מפורש** בכל רשומת כסף (`ILS`) — גם אם היום יש רק מטבע אחד, השדה מונע כאב עתידי.
- **מע"מ** — תחליט אם המחירים המוצגים כוללים מע"מ (לצרכן פרטי בישראל בד"כ כן). שמור גם את שיעור המע"מ על כל חשבונית — שיעורים משתנים עם הזמן וחשבונית היסטורית חייבת לשקף את השיעור שהיה אז.

### 2.6 מחזור חיי העונה (Season Rollover)

עונה ראשונה מקימים מ-0. אבל בכל עונה אחר כך — **לא ממלאים מחדש, מגלגלים ומתקנים:**

- **Rollover** — ביצירת עונה חדשה, המערכת מציעה "להעתיק את מבנה העונה הקודמת?" — כל הקבוצות והשיבוצים מועתקים כבסיס. המועדון רק מתקן הפרשים (מי עזב, הצטרף, עלה קבוצה). זה הופך עבודה של שעות לדקות. אפשרי רק כי הזהות (שחקן/מאמן) מופרדת מהשיוך העונתי.
- **קידום לפי גיל** — בזמן הגלגול, המערכת *מציעה* קידום אוטומטי של שחקנים שעברו את גיל הקבוצה — אבל ההחלטה הסופית בידי האדמין (לפעמים ילד נשאר שנה נוספת).
- **עונה סגורה = read-only** — עונה שעבר זמנה הופכת לקריאה בלבד. מונע טעויות ושומר על אנליטיקה היסטורית נקייה. פתיחה מחדש לתיקון = פעולה מפורשת ומתועדת.

## 3. מודל המשתמשים וההרשאות (RBAC)

### 3.1 רמות המערכת

```
Platform (אתה)
  └── Platform Admin / Support
        │ יוצר ומנהל מועדונים
        ▼
Club (Tenant)
  └── Club Owner / Admin
        │ מנהל הכל בתוך המועדון, יוצר משתמשים
        ├── Staff Roles (מאמן, מזכירות, גזבר...)
        └── (אנשי קשר/הורים = נתונים בלבד, ללא גישה)
```

### 3.2 Roles מוצעים בתוך מועדון

המודל צריך להיות **גמיש (role-based עם permissions גרנולריים)**, לא רשימה קשיחה. אבל כברירת מחדל מומלץ להגדיר:

| Role | גישה טיפוסית |
|---|---|
| **Club Admin** | הכל: משתמשים, עונות, קבוצות, תשלומים, הגדרות, אנליטיקה |
| **Coach / מאמן** | הקבוצות שמשויכות אליו *(לפי `team_coaches`)*: שחקנים, נוכחות, פעילויות. ללא גישה לכספים |
| **Treasurer / גזבר** | תשלומים, חיובים, דוחות כספיים. ללא עריכת הרכב קבוצות |
| **Secretary / מזכירות** | רישום שחקנים ואנשי קשר, פרטי קשר. הרשאות חלקיות לכספים |

> **חשוב: אנשי קשר (הורים/משלמים) אינם משתמשים במערכת.** אין להם login, אין חשבון, אין גישה. הם רשומת *נתונים* בלבד. הדבר היחיד שהם מקבלים הוא קישור תשלום ב-SMS/מייל, שמוביל לעמוד הסליקה של הסולק — ללא התחברות. המערכת סגורה לחלוטין לבעל המועדון ולצוות שהוא בוחר.

**עיקרון מנחה:** הרשאות = רשימת `permissions` (לדוגמה `players.edit`, `payments.view`, `payments.charge`). Role = אוסף permissions. כך אפשר ליצור roles מותאמים אישית לכל מועדון בלי לגעת בקוד.

**כלל אבטחה ברזל:** כל permission נבדק גם מול ה-`club_id` — מאמן של מועדון א' לא יכול לגעת במועדון ב' לעולם, גם אם איכשהו ינחש ID.

### 3.3 Authorization בשכבת ה-API (לא רק RLS)

זו הנקודה הקריטית ביותר באבטחה, ולכן היא סעיף בפני עצמה. **RLS מגן על הנתונים ב-DB; הוא לא מספיק לבדו.** צריך שכבת authorization מפורשת בקוד שנבדקת בכל endpoint:

- **שני שכבות הגנה, לא אחת:** (1) בדיקת permission בקוד לפני כל פעולה (`האם ל-user יש payments.charge?`), ו-(2) RLS ב-DB כרשת ביטחון אחרונה. הראשונה מונעת פעולה לא-מורשית; השנייה מונעת דליפת נתונים אם הראשונה נכשלה.
- **Default deny:** endpoint ללא בדיקת הרשאה מפורשת = חסום. לא ההפך. זה הופך "שכחתי לבדוק" משגיאת-אבטחה-שקטה לכשל-גלוי.
- **בדיקה ברמת הרשומה, לא רק ברמת הסוג:** לא מספיק "מאמן יכול לראות שחקנים" — צריך "מאמן יכול לראות את שחקני *הקבוצות שלו*". הרשאה ברמת ה-resource הספציפי.
- **אותו עיקרון ל-Control Plane:** גישת תמיכה לנתוני מועדון נבדקת, מוגבלת בזמן, ומתועדת ב-audit log.

> **למה זה כאן ולא "אחר כך":** באג ההרשאות הכי נפוץ ב-SaaS הוא endpoint ששכחו להגן עליו. אם "default deny" ו-"בדיקה בכל endpoint" הם עיקרון מההתחלה — לא מתקנים את זה אחרי דליפה.

## 4. אימות והתחברות (Auth)

מערכת מודרנית, RTL, נקייה.

- **הרשמה / התחברות:** דוא"ל + סיסמה, וגם **OTP ב-SMS** (נפוץ מאוד בישראל, נוח להורים).
- **Magic Link** באימייל כאופציה נוספת ידידותית.
- **איפוס סיסמה** מאובטח עם טוקן חד-פעמי בתוקף מוגבל.
- **2FA** אופציונלי לתפקידי Admin/גזבר (נוגעים בכסף).
- **Session management** — טוקנים עם תוקף, refresh, ואפשרות "התנתק מכל המכשירים".
- **הזמנת משתמשים:** Admin מזמין staff במייל → המשתמש מגדיר סיסמה בעצמו. לא יוצרים סיסמאות עבור אחרים.

> **המלצת אבטחה:** אל תאחסן סיסמאות בעצמך מאומה — השתמש ב-hashing חזק (bcrypt/argon2). שקול שימוש בספק Auth מנוהל (כמו Supabase Auth / Auth0 / Clerk) כדי לא להמציא את הגלגל ולחסוך פרצות. זה אחד התחומים שבהם "לבנות לבד" כמעט תמיד טעות.

## 5. ישויות הליבה (Domain Model)

תרשים מושגי של היחסים העיקריים:

```
Club ──< Season ──< Team ──< TeamPlayer >── Player
  │                   │                        │
  │                   └──< TeamCoach >── Coach  └──< PlayerContact >── Contact
  │                                                  (is_billing_contact)
  ├──< StaffMember (עובדים)
  │       └── (קישור אופציונלי ל-) User (login)
  │
  ├──< ClubBranding (לוגו, צבעים, הגדרות תצוגה)
  │
  └──< User (staff login) ──< Role ──< Permission
  │
  └──< Invoice / Payment ── (Contact / Player / Season)
```

> **הבחנה #1 — "אדם" מול "משתמש".** מאמן או עובד הם **אנשים** (people) עם פרטים משלהם — קיימים במערכת גם בלי חשבון התחברות. *User* הוא חשבון login, ורק staff מקבלים אותו. **אנשי קשר (הורים/משלמים) הם נתונים בלבד ולעולם לא מקבלים login** — המערכת סגורה לבעל המועדון ולצוות בלבד.

> **הבחנה #2 — "קשר אנושי" מול "תפקיד חיוב".** במקום ישות "הורה" קשיחה, יש ישות גנרית **`Contact`**. הקישור `PlayerContact` נושא *שני* שדות נפרדים: `relationship` (אבא/אמא/אפוטרופוס/עצמי/אחר) ו-`is_billing_contact` (מי מקבל את קישור התשלום). כך נפתרים בלי כפילויות: הורה שמשלם, שחקן בוגר שמשלם לעצמו (`relationship=self`), סבא שמשלם בעוד ההורה הוא איש קשר לחירום, ושני הורים גרושים. **שחקן בוגר אינו צריך מנגנון נפרד — זה אותו צינור חיוב, רק שהנמען הוא השחקן עצמו.**

### טבלאות עיקריות (תיאור high-level)

- **clubs** — פרטי המועדון, פרטי קשר, הגדרות, סטטוס מנוי.
- **club_branding** — `club_id`, לוגו, צבע ראשי/משני, פייבון, ועוד הגדרות תצוגה. נערך ע"י המועדון באזור ההגדרות (ראה סעיף 5.1).
- **seasons** — `club_id`, שם (למשל "2025/26"), תאריך התחלה/סיום, `is_active`, `status` (פעילה/סגורה).
- **teams** — `club_id`, `season_id`, שם (נערים ב', בוגרים...), קטגוריית גיל.
- **players** — `club_id`, פרטים אישיים, ת.ז., תאריך לידה, `status` (פעיל/לא פעיל/עזב). **קבוע — לא תלוי עונה. לא נמחק כשעוזב.**
- **team_players** — טבלת קישור: `team_id`, `player_id`, `season_id`. כאן חיים השיבוצים העונתיים.
- **coaches** — `club_id`, פרטים אישיים, התמחות, תעודות/הסמכות, רישיון אימון, תאריך תפוגה. ישות קבועה — לא תלוית עונה.
- **team_coaches** — קישור עונתי: `team_id`, `coach_id`, `season_id`, תפקיד (ראשי / עוזר / שוער). מאמן יכול לאמן כמה קבוצות, וקבוצה יכולה להיות עם כמה מאמנים.
- **staff_members** — `club_id`, עובדי המועדון שאינם מאמנים (מנהלן, פיזיותרפיסט, אב-בית...). פרטים, תפקיד, סטטוס. *ישות נפרדת לעתיד — מתוכננת מראש.*
- **contacts** — `club_id`, פרטי איש קשר (שם, טלפון, מייל). ישות נתונים בלבד — **ללא login**. איש קשר אחד יכול להיות מקושר לכמה שחקנים (הורה למשפחה). per-club: אם אותו אדם קיים בשני מועדונים — שתי רשומות נפרדות.
- **player_contacts** — קישור many-to-many בין שחקנים לאנשי קשר, עם `relationship` (אבא/אמא/אפוטרופוס/עצמי/אחר) ו-`is_billing_contact` (boolean). וולידציה: לקטין חייב להיות לפחות איש קשר אחד שאינו השחקן עצמו.
- **users** — חשבון התחברות, `club_id`, role. מקושר אופציונלית ל-staff (coach / staff_member) דרך `person_id` + `person_type`. **אנשי קשר לעולם אינם מקושרים ל-user.**
- **platform_users** — משתמשי ה-Control Plane (אתה). **טבלה נפרדת לגמרי** מ-users של המועדונים.
- **roles / permissions / role_permissions** — מנגנון ה-RBAC.
- **payment_plans** — `club_id`, שם ("דמי עונה", "6 תשלומים"...), סכום (אגורות), מספר תשלומים, סוג (חד-פעמי/תשלומים/מנוי). תבנית שהמועדון מגדיר ובוחר ביצירת חיוב (ראה סעיף 6.4).
- **invoices** — חיוב: סכום (באגורות), מטבע, `payment_plan_id`, מספר תשלומים, נמען (`contact_id`), `player_id`, `season_id`, שיעור מע"מ, סטטוס.
- **payments** — תשלום בפועל מול הסולק: סכום, סטטוס, מזהה עסקה חיצוני (לאימות idempotency), timestamps.
- **audit_logs** — `club_id`, `user_id`, action, entity, before/after, timestamp.

### 5.1 התאמה אישית של המועדון (Branding / White-Label)

כל מועדון יכול להגדיר את המראה של המערכת באזור ההגדרות, בלי מעורבות שלך. זה נשמר בטבלת `club_branding` ונטען דינמית בכל בקשה לפי ה-`club_id`:

- **לוגו** — העלאת קובץ (נשמר ב-object storage כמו S3, לא ב-DB). שמור גם גרסת favicon.
- **צבעים** — צבע ראשי, צבע משני, ואולי צבע הדגשה. נשמרים כ-CSS variables ונטענים בזמן ריצה — כך הממשק כולו (כפתורים, כותרות) מקבל את זהות המועדון.
- **שם תצוגה** — איך המועדון מופיע בכותרות, במיילים ובקישורי התשלום שנשלחים להורים.

**שיקולי מימוש חשובים:**

- **ולידציה על קבצים** — סוג (PNG/SVG/JPG בלבד), גודל מקסימלי, וסניטציה של SVG (SVG יכול להכיל JavaScript זדוני — חובה לנקות).
- **ניגודיות נגישות** — אם המועדון בוחר צבע בהיר על רקע לבן, טקסט ייעלם. שווה לאכוף בדיקת ניגודיות מינימלית או לחשב אוטומטית צבע טקסט קריא.
- **Branding גם במיילים וקישורי תשלום** — כשהורה מקבל SMS/מייל עם קישור תשלום, הלוגו והשם של המועדון אמורים להופיע. זה מה שגורם להורה לתת אמון ולשלם. זו לא רק אסתטיקה — זו המרה.
- **ברירת מחדל נקייה** — מועדון חדש מתחיל עם theme ניטרלי ומקצועי עד שיגדיר משלו.

### 5.2 שני הדשבורדים (Control Plane מול Tenant)

אלה **שני מוצרים נפרדים שעונים על שתי שאלות שונות**, לא אותו דשבורד עם הרשאות שונות. שניהם יושבים על אותו DB אך ניגשים אליו מזוויות הפוכות.

**דשבורד הפלטפורמה (שלך, Control Plane)** — "איך עסק ה-SaaS שלי רץ". רוחבי על כל המועדונים:

- מספר מועדונים פעילים, חדשים, רדומים.
- מי בתקופת ניסיון (trial), מי שילם מנוי ומי בפיגור על המנוי *אליך*.
- הכנסות SaaS, צמיחה, churn.
- מספר שחקנים/משתמשים לכל מועדון (מטריקה מצרפית).
- בריאות מערכת, תורים, שגיאות, פניות תמיכה.

**דשבורד המועדון (Tenant)** — "איך המועדון שלי רץ". מבודד ל-`club_id` ולעונה הנוכחית דרך RLS:

- שחקנים רשומים בעונה, התפלגות לקבוצות.
- כמה כסף נגבה החודש מול צפוי, אנשי קשר בפיגור.
- חיובים פתוחים, תשלומים אחרונים.
- פעילות אחרונה במערכת.

**רמת הגישה שלך לתוך מועדון — החלטה רגישה:** כברירת מחדל אתה רואה רק **מטריקות מצרפיות** ("למועדון א' יש 120 שחקנים"), לא נתונים פנימיים. כניסה לראות שחקן/תשלום ספציפי היא **פעולת תמיכה מפורשת, מוגבלת בזמן ומתועדת ב-audit log** (כמפורט בסעיף 2.4). זה מגן על פרטיות המועדונים ועליך משפטית — קריטי במיוחד עם נתוני קטינים.

## 6. ה-Flow המרכזי: אנשי קשר ותשלומים (החלק הקריטי)

זה הלב של המוצר. ככה הוא עובד מקצה לקצה:

1. **רישום שחקן** — המזכירות/אדמין יוצרים שחקן ומשבצים אותו לקבוצה בעונה הנוכחית.
2. **קישור איש קשר** — יוצרים/מקשרים איש קשר לשחקן (טלפון, מייל), ומסמנים מי ה-`is_billing_contact`. לשחקן בוגר — איש הקשר יכול להיות הוא עצמו (`relationship=self`).
3. **יצירת חיוב (Invoice)** — האדמין/גזבר יוצר חיוב: סכום, סוג (חודשי קבוע / חד-פעמי), עונה. אפשר לאגד חיובים של כמה ילדים לאותו איש קשר משלם.
4. **שליחת קישור תשלום** — המערכת שולחת ל-billing contact **קישור תשלום ייעודי ומאובטח** (SMS / מייל, עם branding המועדון) לעמוד סליקה מתארח של ספק הסליקה.
5. **תשלום** — המשלם משלם בעמוד מאובטח של הסולק (PCI מטופל על ידם — אתה לא נוגע בפרטי הכרטיס).
6. **Webhook** — הסולק שולח חזרה אישור → המערכת מעדכנת את ה-`payment` ו-`invoice` ל"שולם", מייצרת חשבונית, ושולחת קבלה.

### 6.1 שיקולי סליקה (ישראל — Tranzila / Cardcom / Meshulam)

> **חשוב — תחום שצריך זהירות והוא תלוי בחירת ספק:**

- **Hosted Payment Page** — תמיד להעדיף עמוד תשלום מתארח של הסולק (redirect / iframe) ולא לאסוף פרטי כרטיס בעצמך. כך אתה **מחוץ ל-scope של PCI-DSS** ברובו — חוסך עלויות תאימות אדירות וסיכון משפטי.
- **Tokenization לחיובים חוזרים** — לחיוב חודשי אוטומטי, הסולק מחזיר *token* של הכרטיס (לא המספר עצמו). אתה שומר רק את ה-token ומחייב מולו. בדוק שהספק תומך ב-recurring/tokenization.
- **Webhooks + אימות חתימה** — חובה לאמת שה-webhook הגיע באמת מהסולק (חתימה/secret), אחרת מישהו יכול לזייף "שולם".
- **Idempotency** — webhook יכול להגיע פעמיים. כל עיבוד תשלום חייב להיות idempotent (לפי מזהה עסקה ייחודי) כדי לא לסמן/לחייב פעמיים.
- **חשבוניות ומע"מ** — בישראל יש דרישות לחשבונית מס. חלק מהסולקים (Cardcom) מנפיקים חשבוניות מובנה; אחרת צריך אינטגרציה למערכת חשבוניות (כמו iCount / Green Invoice). **כדאי לבדוק את זה מול הספק לפני בחירה — זה יכול לחסוך פיתוח שלם.**
- **Reconciliation** — מסך שמראה לגזבר: מי שילם, מי בפיגור, סכומים פתוחים. זה ה-ROI האמיתי של המערכת עבור המועדון.

> **המלצה לבחירת ספק:** אם נפיקת חשבוניות מובנית חשובה לך — **Cardcom** חזק שם. אם אתה רוצה חוויית מפתח/API נוחה יותר ותמיכה טובה ב-tokenization — שווה להשוות **Meshulam (גרואו)** ו-**Tranzila**. ההמלצה: בחר לפי (1) תמיכה ב-recurring tokenization, (2) חשבוניות, (3) איכות ה-API והתיעוד.

### 6.2 מודל הסטטוסים של חיוב (State Machine)

"שולם / לא שולם" הוא בינארי נאיבי שיישבר מול המציאות. הגדר את מצבי החיוב מראש — שינוי מבנה אחרי שיש כסף אמיתי במערכת הוא הזמן הכי גרוע:

| סטטוס | משמעות |
|---|---|
| `draft` | חיוב נוצר, טרם נשלח |
| `pending` | קישור נשלח, ממתין לתשלום |
| `paid` | שולם במלואו |
| `partially_paid` | שולם חלקית (למשל תשלום ראשון מתוך כמה) |
| `failed` | ניסיון תשלום נכשל |
| `refunded` | הוחזר |
| `canceled` | בוטל לפני תשלום |
| `overdue` | עבר תאריך יעד ללא תשלום |

מעברים מותרים בלבד (לא כל מצב לכל מצב), וכל מעבר נכתב ל-audit log. זה מאפשר דוחות אמינים לגזבר ומונע מצבים "תקועים".

### 6.3 ישויות אמינות תשלומים (לא רק עקרונות)

שני אלה צריכים להיות **טבלאות**, לא כוונות טובות:

- **payment_events / webhook_log** — כל webhook נכנס נשמר *לפני* עיבוד: raw payload, מזהה אירוע ייחודי מהסולק, סטטוס עיבוד. webhook שכבר עובד (לפי המזהה) — מדולג. כך חיוב כפול הופך לבלתי אפשרי, ויש לך עקבות מלאים לדיבוג מול הסולק.
- **idempotency_keys** — כל פעולת כסף יוצרת מפתח ייחודי. בקשה חוזרת עם אותו מפתח מחזירה את התוצאה הקודמת במקום לבצע שוב. מגן גם מפני לחיצה כפולה של משתמש וגם מפני retry של רשת.

### 6.4 מודלי תשלום, תשלומים ומנוי

המערכת תומכת בכמה מודלי תשלום שהמועדון מגדיר. **"מודל תשלום" הוא תבנית של פרמטרים** שקובעת מה נשלח לסולק בעת יצירת הקישור — אותו צינור בדיוק, רק נתונים שונים בבקשה.

טבלה חדשה: **payment_plans** — `club_id`, שם ("דמי עונה מלאים", "6 תשלומים", "דמי הרשמה"), סכום (אגורות), מספר תשלומים, סוג. החיוב (invoice) מצביע על ה-plan שנבחר. הגזבר בוחר plan ביצירת החיוב, והקישור נבנה בהתאם.

**שלוש דרכי גבייה — שלוש רמות מורכבות:**

| מודל | מי מנהל את הפיצול | מורכבות אצלך |
|---|---|---|
| **חד-פעמי** | — | אפסית |
| **תשלומים (6/10/12)** | הסולק, בעסקה אחת | נמוכה — שולחים "סכום + מספר תשלומים", מקבלים webhook אחד |
| **מנוי חודשי מתחדש** | תלוי בבחירה (ראה למטה) | בינונית-גבוהה |

**נקודה קריטית על תשלומים:** את הפיצול ל-6/10/12 תשלומים מנהל **הסולק**, לא אתה. אתה שולח "1,200₪, 6 תשלומים", הסולק מציג להורה עמוד שמפצל ומחייב את הכרטיס לפי הסדר, ומחזיר webhook אחד. במערכת שלך זו נשארת **רשומת חיוב אחת** — אינך מנהל לוח תשלומים ולא רודף אחרי כל תשלום בנפרד.

**הבחנה — מנוי חודשי מתחדש (recurring):** זה שונה מ"תשלומים". מנוי מתחדש כל חודש ללא הגבלה, ויש שתי דרכים:

- **קישור חוזר** — כל חודש המערכת שולחת קישור חדש שההורה לוחץ ומשלם. פשוט למימוש, אבל תלוי בהורה שיזכור (יותר פיגורים).
- **חיוב אוטומטי (token)** — ההורה מאשר פעם אחת, והמערכת מחייבת כל חודש מול ה-token השמור אצל הסולק (לא מספר הכרטיס). נוח, פחות פיגורים — אך דורש שהסולק תומך ב-recurring tokenization והסכמת ההורה מראש לחיוב חוזר. מנגנון: job מתוזמן יוצר חיוב חודשי חדש ומחייב מול ה-token.

> **המלצה:** התחל בלי מנוי-מתחדש בכלל. חד-פעמי + תשלומים מכסים את רוב הצרכים ומורכבותם נמוכה. הוסף recurring (אם בכלל) כשתהיה דרישה אמיתית — המודל כבר ערוך לקלוט אותו.

## 7. שלבי פיתוח (Roadmap)

לפי הדירוג שלך. כל phase נותן ערך עצמאי.

### Phase 0 — תשתית

- בחירת stack, הקמת DB עם RLS, סכמת multi-tenant, CI/CD, סביבות (dev/staging/prod).
- Auth בסיסי (מייל+סיסמה) + שכבת authorization מפורשת (default-deny, סעיף 3.3).
- Platform Admin מינימלי: יצירת מועדון + Club Admin ראשון + דשבורד פלטפורמה בסיסי (סעיף 5.2).
- תשתית בסיסית: audit log, soft-delete, queue ל-jobs, שכבת notifications.
- **Observability + alerting + rate limiting מהיום הראשון** (סעיף 2.3): structured logging, error tracking, health checks, התראות על כשלי תשלום/webhook/זמינות, והגבלת קצב על endpoints רגישים.
- גיבויים אוטומטיים מהיום הראשון.

### Phase 1 — ליבה תפעולית *(העדיפות #1 שלך)*

- עונות: יצירה, הפעלה, פילטור גלובלי לפי עונה.
- קבוצות: CRUD, קטגוריות גיל.
- שחקנים: CRUD, שיבוץ לקבוצות בעונה, סטטוסים.
- **מאמנים: CRUD (פרטים, הסמכות), שיוך עונתי לקבוצות.**
- **גלגול עונה (Season Rollover): העתקת מבנה העונה הקודמת + תיקון הפרשים (ראה סעיף 2.2).**
- **התאמת branding: לוגו, צבעים, שם תצוגה באזור הגדרות (סעיף 5.1).**
- דשבורד מועדון בסיסי (Tenant, סעיף 5.2).

### Phase 2 — אנשי קשר ותשלומים *(העדיפות #2 — ה"וואו")*

- ניהול אנשי קשר (`Contact`) + קישור לשחקנים, סימון billing contact.
- הגדרת מודלי תשלום (`payment_plans`): חד-פעמי + תשלומים.
- יצירת חיובים (לפי plan), אגד חיובי משפחה.
- אינטגרציית סליקה: hosted page, webhook + אימות חתימה, idempotency.
- שליחת קישורי תשלום ב-SMS/מייל (עם branding המועדון).
- מסך reconciliation לגזבר + חשבוניות.
- *(מאוחר יותר, לפי דרישה: מנוי חודשי מתחדש עם token).*

### Phase 4 — אנליטיקה ותוספות *(עדיפות #4)*

## 8. ה-Stack שנבחר

**הוכרע: Supabase + Next.js.** הבחירה נובעת ישירות מהצרכים במסמך — RLS לבידוד מועדונים, Auth מנוהל, ומהירות ל-MVP — לא מהיותה "המתקדמת ביותר".

| שכבה | בחירה | למה |
|---|---|---|
| DB | **PostgreSQL (דרך Supabase)** | RLS אזרח ממדרגה ראשונה — בדיוק מודל הבידוד שבנינו |
| Auth | **Supabase Auth** | מייל+סיסמה, OTP ב-SMS, magic links, sessions — בלי לכתוב קוד אבטחה רגיש |
| Frontend | **Next.js + React** | SSR, RTL נוח לעברית, אקוסיסטם עשיר |
| UI | **Tailwind + shadcn/ui** | רכיבים נקיים, תמיכת RTL טובה |
| Storage | **Supabase Storage** | לוגואים של ה-branding (S3-compatible) |
| לוגיקת כסף | **Supabase Edge Functions** (או Next.js API routes) | webhooks, idempotency, מנוע חיובים — שכבת שירות אמיתית לכסף |
| Queue/Jobs | **Supabase + cron** (או Redis+BullMQ בהמשך) | SMS, מיילים, תזכורות, חיובים מתוזמנים |
| Hosting | **Vercel** (frontend) + **Supabase** (backend) | פשטות תפעולית, deploy מהיר |

> **הערה על צמיחה:** ככל שלוגיקת הכסף תגדל, שכבת ה-Edge Functions תתפוס יותר משקל — אפשר אף להוסיף backend ייעודי (NestJS) ל-Phase 2 בלי לשבור את הארכיטקטורה. מתחילים פשוט, מרחיבים לפי צורך.

## 9. סיכונים ונקודות תשומת לב

1. **דליפת נתונים בין מועדונים** — הסיכון מספר 1 ב-SaaS multi-tenant. RLS + authorization בשכבת ה-API (סעיף 3.3) הם שני קווי ההגנה. כתוב בדיקות אוטומטיות שמוודאות שמועדון א' *לא* רואה נתוני ב'.
2. **endpoint לא מוגן** — באג ההרשאות הנפוץ ביותר. עיקרון default-deny + בדיקה בכל endpoint (סעיף 3.3).
3. **חיוב כפול / טעויות כסף** — idempotency_keys + webhook_log + audit log + אימות חתימות.
4. **תאימות PCI** — אל תיגע בפרטי כרטיס. hosted page בלבד.
5. **ביצועים בסקייל** — אינדקסים נכונים על `club_id` ו-`season_id` בכל שאילתה.
6. **פרטיות קטינים (חוק הגנת הפרטיות)** — soft-delete, מנגנון מחיקה לפי בקשה, מינימום נתונים, retention מוגדר.
7. **גיבויים ושחזור** — DB מנוהל עם backups אוטומטיים יומיים + point-in-time recovery. בדוק *שחזור* בפועל מדי פעם — גיבוי שלא נבדק אינו גיבוי. הגדר RPO/RTO (כמה נתונים מותר לאבד / כמה זמן לחזור לאוויר).
